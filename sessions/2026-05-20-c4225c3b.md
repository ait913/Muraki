## 課題点

- 設計判断を 1 回ひっくり返した (443 ルート→80 ルート)
  - **何が課題か**: 「Nginx 無変更で 443 ルート」をユーザー承認まで取って実装、疎通確認まで完了させてから「80 化したい」と判断が変わった
  - **いつ起こるか**: 設計時に「最小変更」と「クリーン形」のトレードオフを明示しきれず、片側を提案して通った後で本人の哲学 (ミニマル/明示的) との不整合に気付くケース
  - **次どうするか**: 設計提示時に複数オプション (今すぐ完了 / 完成形まで一気にやる / 段階的) のテーブルを最初から出す。本セッションでは Touri が後で気付いて方針変更した形になったが、最初に Touri 流が出てればやり直し不要だった

- `cloudflared tunnel login` の 1-zone 制限を事前察知できなかった
  - **何が課題か**: Researcher のリサーチ範囲に「複数 zone 認可」のテストが含まれず、login → tunnel route dns で wildcard を 3 zone 分まとめて作る前提で動いて誤 CNAME を 3 個作ってしまった
  - **いつ起こるか**: Cloudflared CLI を複数 zone 環境で初めて使うとき
  - **次どうするか**: knowledge `[[tool-quirk/cloudflared-tunnel-login-multi-zone]]` を残したので次回は事前回避できる。新規ツール検証時は「単一資源 vs 複数資源」のテストを必ず研究項目に入れる

- Cloudflare Access の One-time PIN がメール届かず詰まった
  - **何が課題か**: Application で OTP を選択しても Zero Trust → Integrations → Identity providers で OTP を enable しないと送信されない (旧 UI と新 UI で経路が違う、Cloudflare docs も古い)
  - **いつ起こるか**: Access Application を最初に作成するとき (誰でも初回はハマる)
  - **次どうするか**: `[[library/cloudflare-tunnel-2026]]` の SSH セクションに「OTP は Integrations 配下で enable」を明記済。Application 単位の選択と global の enable は別系列、と覚える
