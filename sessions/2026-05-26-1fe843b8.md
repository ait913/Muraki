  - **いつ起こるか**: Figma board / file を chrome-devtools MCP で取得しようとした全てのケース
  - **次どうするか**: Figma 内容が必要なときは最初から Touri にスクショ取得依頼。MCP で取りに行く時間が無駄。memory or knowledge に Figma=スクショ依頼デフォルトの注記検討

  - **何が課題か**: Notion MCP 認証 URL を Touri に投げたが、結局 Notion ページが public 共有だったので chrome-devtools MCP で取れた。並列で頼んだが Touri が踏まないままでも進めた
  - **いつ起こるか**: Notion ページが public 共有のとき、認証要らず chrome-devtools MCP で snapshot を取れば足りる
  - **次どうするか**: Notion URL もらったらまず chrome-devtools MCP で開いて public 判定。認証 URL はその後の fallback として残す

- **researcher の version 情報を Codex がそのまま package.json にコピーした (架空 version で install 全 abort)**
  - **何が課題か**: researcher が「2026-05-26 最新」リストとして書いた `@tanstack/router-plugin@1.170.8` が npm 上に**存在しなかった** (実最新は `1.168.11`)。Codex は research summary を信用して package.json に転記、`npm install` が ETARGET で全 abort → pre-flight check で初めて発覚
  - **いつ起こるか**: researcher が新規 PJ で「最新 version listing」を書くとき。peer/関連 package を「同じ系列だろう」と推測した瞬間にズレる
  - **次どうするか**: researcher prompt に「package version は **必ず `npm view <pkg> version` で個別実機確認**、推測列挙禁止」を明示。Leader の pre-flight check で全パッケージの version 実在を git commit 前に確認する

- **Coolify private repo で deploy が初手で失敗 (SKILL に明示なし)**
  - **何が課題か**: `gh repo create` のデフォが private、`appily` SKILL の「新規アプリ作成標準フロー」が public repo 前提なので、初回 deploy で `git ls-remote → could not read Username` 即 fail
  - **いつ起こるか**: 新規 GitHub repo を private で作って Coolify に load する瞬間
  - **次どうするか**: gotcha `coolify-private-repo-cannot-clone.md` 追加済。次回新規アプリは `gh repo view --json visibility` を最初にやる。`appily` SKILL の Step 0 として「repo visibility 確認 → public でなければ public化 / GitHub App / Deploy Key の三択」を追加候補

- **Stale label loop は新規 app でも普通に発生する (1 発で 200 にならない)**
  - **何が課題か**: SKILL の「is_force_https_enabled=false 必須」を守っても、新規 Dockerfile app で `https://*.appily.run/` が self-redirect ループ。fqdn drop→restore で復旧
  - **いつ起こるか**: 新規 Coolify Dockerfile app の初回デプロイ後、毎回ではないが 2-3 回に 1 回
  - **次どうするか**: 本番デプロイのワークフローに「Step 5 確認で 307/302 self-redirect なら即 drop/restore リカバリ」を**ハードコード**。SKILL 末尾の note では弱い、新規 app 作成標準フローの一部として組み込む候補

- **chrome-devtools MCP の fill が React controlled input の onChange を発火させない**
  - **何が課題か**: 実機検証で `/create` の表示名 input に MCP `fill` で値を入れて「つぎへ」を click → submit せず。a11y snapshot の value は更新されるが React state は `""` のまま、`disabled` が外れていなかった
  - **いつ起こるか**: chrome-devtools MCP で React 19 controlled input を操作する全てのケース
  - **次どうするか**: gotcha `chrome-devtools-mcp-fill-react-controlled-input.md` 追加済。実機検証では最初から **API 直叩き seed** をデフォルト戦略にする (form 経由を避ける)。MCP fill は表示確認専用
