## 課題点

1. **設計初稿のスコープ縮小**: Architect がデータ源の制約 (出席集計のみ) を見て要望を満たさない縮小版を出した。「多すぎる場合は省略」という要望文言が複数イベント前提なことを読めば縮小は避けられた。いつ起こるか=データ源が一見足りない時。次どうするか=要望の含意 (件数・動作) からあるべきデータ量を逆算し、既存ユーティリティ (meetingExpansion) で埋められないか先に当たる。
2. **描画テストの偽陰性 YELLOW で 1 往復増えた**: 設計doc にコンポーネント公開 prop 契約が無く、Reviewer が prop 形を誤って全 assert 落ち。いつ起こるか=Reviewer に描画テストを書かせる項目。次どうするか=Architect が render テスト対象の公開 prop 型 (フィールド名・形・コールバック引数) を設計doc に最初から転記する ([[gotcha/design-must-specify-component-prop-contract-for-render-tests]] 化済)。
3. **サブエージェントの background 委譲で成果未回収**: developer/reviewer が重い codex 生成をバックグラウンドに投げてターンを返し、生成物が消えかけた (2回)。いつ=Codex に大きな生成を投げる時。次=召集プロンプトに「同期実行・バックグラウンド委譲禁止」を明記する。
4. **同一 YELLOW を反復 (validation error.code)**: zValidator が生 ZodError を返すのに Reviewer が `error.code==="VALIDATION_ERROR"` を strict assert する偽陰性を Phase2/曜日 で再度踏んだ (gotcha 既存だが運用に乗り切れず)。次=召集時に「validation は status 中心で assert」を毎回明記。
5. **Chrome captureScreenshot が恒常タイムアウト**: 長時間セッションで `take_screenshot` が連続失敗 (snapshot/evaluate は可)。次=Web ビジュアルは `evaluate_script` の computed/innerHTML 実測を主・画像は補助。iOS は `simctl io screenshot` が安定動作。

## いい点

1. **Leader の実装 Read による YELLOW 切り分けが効いた**: Reviewer の「実バグか不明」を、TimetableView を 1 ファイル読むだけで「実装は正しく契約欠落が原因」と確定でき、設計doc 補強 → 再テスト GREEN に最短で着地。CLAUDE.md の「YELLOW/RED 時のみ該当 diff を読む」例外運用が機能。続け方=判定が割れたら推測で再実装させず、まず公開インターフェースだけ確認する。
2. **キャラ生成を冒頭でバックグラウンド並行着手**: 独立タスク (Codex imagegen) を設計フェーズと並走させ、承認時には候補が揃っていた。続け方=生成系・独立アセットは早期に background 投入。
3. **git 状態の事前検証**: feature/mvp と main の lineage を merge-base で確認し、陳腐化ブランチを掴まずに済んだ。続け方=worktree を切る前にブランチ関係と未追跡ファイルの配線を必ず確認。
