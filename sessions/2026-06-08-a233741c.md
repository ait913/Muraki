## 課題点

- chrome-devtools MCP の profile ロック衝突
  - **何が課題か**: GUI Chrome (chrome-login.sh 由来) が userDataDir を握ったままで MCP headless が起動できず `navigate_page` がエラー
  - **いつ起こるか**: 過去セッションのログイン用 GUI Chrome を閉じ忘れている時
  - **次どうするか**: `pkill -f "chrome-devtools-mcp/chrome-profile"` で headless を解放してから navigate (knowledge/tool-quirk/chrome-for-testing.md の手順通り)

- Coolify 新規アプリの 307 self-redirect loop が `is_force_https_enabled=false`+`redirect:both` でも残った
  - **何が課題か**: 既知の Coolify/Traefik stale label bug。PATCH だけでは proxy label が再生成されない
  - **いつ起こるか**: Cloudflare 配下の Coolify で新規アプリを立てた直後
  - **次どうするか**: fqdn 削除→redeploy(404で Origin 起因確定)→戻し→redeploy の label 再生成を最初から手順に織り込む ([[gotcha/coolify-traefik-stale-label-loop]])。なお Coolify の logs API は Node アプリで空を返すことがあり、SMTP 等の挙動確認は SSH か外形で取る

- 視覚バグ (白地白文字ボタン) は自動テストで拾えない
  - **何が課題か**: `.primary` の背景が詳細度負けして white になっても、data-testid/prop テストは全 GREEN。computed-style に依存しない単体テストの死角
  - **いつ起こるか**: CSS 詳細度競合 (包括セレクタ `.toolbar button` vs `.primary`) が起きた restyle 時
  - **次どうするか**: UI 再設計は必ず Leader が実機 (ローカルログイン+ダミーデータ) で desktop/mobile×light/dark スクショ確認するのを手順化。テストは「壊れてない」は言えても「綺麗」は言えない

- 未来出費アプリの **pre-anchor 月で `/api/months` が 500** (既存 backend エッジ)
  - **何が課題か**: anchor 月より前へ月送りすると 500。MonthView がエラーを graceful に扱わない可能性
  - **いつ起こるか**: ユーザーが基準残高設定日より前の月を開いた時
  - **次どうするか**: backend で pre-anchor 月を空 or clamp 応答にする + MonthView の query error ハンドリングを次イテレーションで (今回 UI 再設計スコープ外)
