## 課題点

- **developer subagent が Codex をバックグラウンド起動して「監視タスク設置」とだけ報告し終了する事故**: 何が課題か = subagent 終了で Codex が孤児化し実装が途中で止まる。いつ起こるか = 長時間 Codex 実行を developer が background + Monitor で待とうとした時。次どうするか = developer 召集プロンプトに「Codex 完了まで待ってから報告 (バックグラウンド放置禁止)」を毎回明記する (今回の後任召集で有効だった)
- **ベースライン失敗テストの山が「既存破損 = 本番バグ」を隠していた**: 何が課題か = api 21件のうち stats 5件は実は本番バグ (出欠ルール無視) のシグナルだったのに「ベースライン失敗」として除外され続け、再設計マージ時点で見逃された。いつ起こるか = ベースライン失敗を機械的に判定除外する時。次どうするか = ベースライン失敗を「テストが古い/環境(localStorage等)/実装バグ」に分類する棚卸しを別タスクで必須化する。今回 stats 5件は解消したが、残る api 16件 (auth/friendship/room系) + web routes 27件 (jsdom localStorage) は未分類のまま放置中 → 次セッションで切り分ける
- **再設計時にテスト全体を GREEN にせずマージした**: 何が課題か = 20260611-semester-redesign マージ時、stats 5件失敗を「ベースライン」と判断してそのまま通したが、実際は再設計が触った領域の隣接バグだった。いつ起こるか = 大きめの再設計で既存失敗を十分精査せずマージする時。次どうするか = マージ前に「失敗テストが触った領域と無関係か」を1件ずつ確認する
- **マージ後の main で prisma generate / 残骸ファイルにより build・typecheck が一時失敗**: 何が課題か = worktree では通るが main では環境差 (stale Prisma client、untracked 残骸) で落ちる。いつ起こるか = schema 変更を含むブランチのマージ後。次どうするか = マージ後検証手順に「prisma generate → build → typecheck」を定型化する

## いい点

- **researcher / architect / developer / reviewer 各 2 体の並列召集が全段で機能**: 何が良かったか = 2 機能を同一壁時計時間で完遂、スコープ境界 (semester/ は②専属) を設計 doc 冒頭に明記したことでマージがコンフリクトゼロ。いつ効いたか = 独立性の高い 2 トラック構成にした時。どう続けるか = 競合しそうな共有ファイル (routes/attendance.ts 等) は「片方は追記のみ」と境界を doc に書かせる
- **reviewer が偽 RED を実装を見ずにプローブで切り分け**: 何が良かったか = bulk API 全滅に見えた失敗を fixture の日付規約ミスと特定し、実装は正しいと証明。いつ効いたか = テストと実装の不一致が出た時。どう続けるか = reviewer に「失敗時は実装でなく本番経路の観察で切り分ける」姿勢を維持させる (gotcha 化済み)
- **設計 doc の導出値誤記を規範ルールで解消する判断**: 何が良かったか = doc 内の「規則」と「手計算の合計」が矛盾した時、規則を規範としてテストを直し、doc は errata 訂正。いつ効いたか = ①の aria-hidden 数、②の rateColor 例。どう続けるか = architect に「導出値は規則から機械的に出す (手計算を書かない)」を浸透させる (gotcha 化済み)
- **「既存破損」報告を鵜呑みにせず待ち時間に切り分けて本番バグを掘り当てた**: 何が良かったか = developer/reviewer が「stats 5件はベースライン」と言ったのを、デプロイ待ちの空き時間に researcher で切り分けたら本番の出席率誤計算バグだった。いつ効いたか = 「既存失敗だから無視」で流しかけた時。どう続けるか = ベースライン失敗を「無視してよい」と確定する前に、最低限どの領域の失敗かを一度は調べる
- **negative control でテストの検証力を確認**: 何が良かったか = reviewer が修正前 src に戻して「テストがバグ状態を捕捉できる」ことを確認してから GREEN を出した。いつ効いたか = 「テストが実装に迎合していないか」が疑わしいバグ修正で。どう続けるか = バグ修正系のレビューでは negative control を標準にする
