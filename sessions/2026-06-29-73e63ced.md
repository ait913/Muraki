## 課題点

- 初版設計が過剰（マルチテナント+DB+ダッシュボード）。Touri のスコープ縮小で救われた。
  - **何が課題か**: MVP スコープが固まりきる前に Architect に投げた
  - **いつ起こるか**: 要件確認より先に「作れる形」を設計させた時
  - **次どうするか**: 承認ゲート前に Leader が「最小/ステートレスで閉じるか」の問いを立ててから Architect 召集
- テストが in-process 型渡しで、実ホストの JSON schema 経路(配列マーシャリング)を検証してなかった。
  - **何が課題か**: Reviewer GREEN でも実ホスト経路の穴(配列文字列化疑い)は残った
  - **いつ起こるか**: MCP ツールを in-process ClientSession だけでテストした時
  - **次どうするか**: 重要ツールは自前mint token + go-sdk client の**実プロトコルスモーク**を標準化（本セッションで手法確立済）
- ステートレス設計が agent の直感(取り込みステップ)に反し submit_plan 幻覚を招いた。
  - **何が課題か**: 「agent が何を期待するか」視点が設計時に抜けた
  - **いつ起こるか**: 慣習的なツール名(submit_*)を消した時
  - **次どうするか**: 消したツールは rubric/description で明示的に「存在しない」と先回り
- 秘密(client secret)が chat 露出。UI投入を勧めたが結局 API 投入に。
  - **次どうするか**: 露出後は regenerate を必須TODO化。最初から Coolify UI 投入を強制する導線
- **「旧App引退していい」の指示が曖昧で、Coolifyクローン用GitHub App(4215780)まで誤削除 → デプロイ不能(Integration not found) → app作り直しの大回り**。
  - **何が課題か**: 同種リソースが複数(認証用App 4213909 / クローン用App 4215780)ある状態で「旧Appは消していい」と役割/IDを曖昧に言った
  - **いつ起こるか**: 破壊操作を役割名だけで指示した時(どっちのAppか特定されてない)
  - **次どうするか**: 破壊操作の指示は必ず**具体ID+役割**を明記。特に**Coolify等インフラに紐づくものは「消すな」を先に**言う
- **1セッションで方針が大きく3転**(stateless → B(OAuth) → stateful マルチテナント)。都度 設計/実装まで走って後で反転した。
  - **何が課題か**: プロダクトの根本要件(ダッシュボード要否・誰が何を見る/操作するか・認証摩擦の許容)が固まる前に技術設計に入った
  - **いつ起こるか**: 「作れる形」を先に設計し、要件を後で詰めた時
  - **次どうするか**: **「誰が何を見る/操作するか」を最初に確定**してから設計。特に**ダッシュボード有無=DB有無=アーキ全体**が決まるので、そこを最初の問いにする。技術設計はその後
