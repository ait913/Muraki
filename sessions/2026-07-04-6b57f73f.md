## 課題点

- **known-failures 台帳はまだ規約だけで実体ゼロ**: 何が課題か = atender の api 16 件 + web 27 件の未分類失敗が台帳なしのまま。いつ起こるか = 次に atender を触ってマージ判断するとき、規約上「未分類のままマージ不可」に引っかかる。次どうするか = 次回 atender セッションの冒頭で棚卸しして `.knowledge/known-failures.md` を作る (6/11 セッションの積み残しと同一タスク)
- **エスカレーションシグナルの網羅性は未検証**: 何が課題か = シグナル 5 種は過去 4 セッションの事故から逆算したもので、網から漏れる異常は Opus Leader では素通りするリスクを (a) 採択時に明示的に受け入れた。いつ起こるか = 未知パターンの障害時。次どうするか = Opus 運用で「シグナルに引っかからなかったが Touri 判断が必要だった」事例が出るたびシグナル定義を置換更新する
- **researcher の「デフォルト = Gemini」が現実と乖離**: 何が課題か = gemini CLI が IneligibleTierError (free-tier 廃止) で死んでおり、agent 定義の第一経路が使えない。今回は習慣ノートのフォールバック記述 (WebSearch/WebFetch/codex) で回ったが、定義と実態のズレが残っている。いつ起こるか = researcher 召集のたび。次どうするか = gemini の復旧 (Antigravity 移行 or 課金) を確認し、恒久なら researcher.md の道具の使い分けを置換更新する

## いい点

- **設計思想を先に聞いてから提案を仕分けた**: 何が良かったか = 初回提案の「ルール化」一辺倒を、Touri の背景 (エキスパート制の意図) を受けて手順/ツーリング/経験の 3 分類に再精査し、思想と両立する形だけ残せた。いつ効いたか = 提案の 2 巡目。どう続けるか = 組織系の変更は「なぜ今の形か」を先に確認してから設計する
- **習慣ノートを「実際に踏んだ穴 + セッション出典」に限定した**: 何が良かったか = 予防的な網羅ルールを書き始めるとルール組織に退行する。出典必須にしたことで肥大化に歯止め。どう続けるか = ノート追記時は必ず sessions/ の実績を引く
- **習慣ノートの自己更新が導入翌日に初稼働**: 何が良かったか = researcher が HIG の WebFetch ハルシネーション罠を発見 → 指示なしで自分の習慣ノートに追記 + 手法を library 化。「モデルが変わっても制度的記憶が残る」設計が実際に回った。いつ効いたか = デザイン観点リサーチ中。どう続けるか = 召集プロンプトの冒頭に習慣ノート必読を書き続ける (agent 定義配線だけに頼らない)
- **一次ソース主義が HIG の SPA 化を突破した**: 何が良かったか = WebFetch が空振り/ハルシネーションでも諦めず docs JSON エンドポイントを見つけて 14 ページ完全取得。おかげで「8pt グリッドは HIG に無い」「タブ 3-5 は旧世代」という要約サイト経由では得られない確定が取れ、既存ナレッジの誤記まで修正できた。どう続けるか = ガイドライン系の調査は必ず原文取得 (要約記事出典を許さない)
